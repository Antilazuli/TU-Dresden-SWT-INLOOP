Linked List

[my solutions]

The Java standard library comes with a lot of powerful container classes (also known as the Collections Framework) such as java.util.ArrayList. However, in this lesson you are going to develop your own implementation of a linked list.

Your task is to write a singly linked list without using the collections framework. To keep things simple, elements of this list only hold objects of type String.

You need to implement two classes:

  • ListElement, which represents one element of the list.
  • List, which represents a linked list itself.

Another important part of this task is proper handling of invalid method arguments. The last section contains hints on how to achieve this.

The ListElement class

Each ListElement has an attribute content of type String representing the element’s content. Furthermore, it has an attribute next that references the successor element in the list. It should be set to null if there is no successor. The constructor of ListElement has one parameter content of type String, which is used to initialize the attribute of the same name. It must ensure that the ListElement cannot be empty, i.e., that content cannot be null or the empty string.

The attributes must have a visibility of private. The class must have public getter methods (getContent() / getNext()) and public setter methods (setContent(…) / setNext(…)) to make it possible to access or alter the value of the attributes from outside. The return type of setter methods is always void.

Ensure that a ListElement cannot be made empty at any point by throwing an IllegalArgumentException at the appropriate locations.

The List class

First of all, List has a head attribute that contains the first ListElement of the list or null in case the list is empty. This list head is the beginning of the chain of ListElements.

It must be possible to append a new element to the end of the list with the method append(…), which takes one argument of type String. This argument is the content of the new element.

Also, it must be possible to remove an element with the method remove(…), which takes one argument of type String. It must remove the first element whose content equals the given argument and return the removed element’s content or null if no element was removed.

The argument of both append(…) and remove(…) must not be null or the empty string. The methods should throw an IllegalArgumentException if this rule is violated.

Correctly maintaining the links of the resulting chain can be tricky! Be careful to think about the special cases of an empty list and a list with only one element.

Hint: Handling invalid method arguments

Whenever you write a method that takes arguments, you should think about the possible values that can be passed to it and ask yourself: which of these inputs shouldn’t be allowed? For example, 25 is not a valid hour, 13 is not a valid month, -1 is not a valid array index, and so on. Or as specified above, it should be impossible to create empty list elements.

As a Java programmer, you can enforce such rules by checking the argument values and throwing an exception if the rules are violated. This is part of an important programming technique known as defensive programming.

Handling invalid method arguments in Java

More specifically, in Java you throw exceptions of the class IllegalArgumentException. According to the documentation of IllegalArgumentException, it is “thrown to indicate that a method has been passed an illegal or inappropriate argument”. For example, the following ListElement constructor ensures that this.content cannot be initialized to null:

public ListElement(String content) {
    if (content == null) {
        throw new IllegalArgumentException("content must not be null");
    }
    this.content = content;
}

(Note that the null check is not the only check that you need to perform in this task – you need to figure out the other one on your own.)

As you can see, the constructor of the exception takes a string as argument. It can be used to pass a message, such as the reason for the exception. You should always provide a meaningful message, because this makes debugging easier.

For example, assume we call the constructor like this:

ListElement element = new ListElement(null);

Then our program will exit with an error message that contains the reason and helps us to find the source of the error:

Exception in thread "main" java.lang.IllegalArgumentException: content must not be null
        at ListElement.<init>(ListElement.java:9)
        at MyListTest.main(MyListTest.java:3)

(<init> is the Java internal name of the constructor method.)

By the way…

In the past, it was considered best practice to throw a NullPointerException (NPE) instead of an IllegalArgumentException (IAE) in this specific case. For example, you’ll find that some parts of the Java Standard Library use NPEs instead of IAEs to indicate that an inappropriate argument of null was passed to a method.

Previous versions of this task also expected NPEs instead of IAEs in such cases. This is no longer the case as of summer term 2020. It now uses IAEs consistently to indicate invalid method arguments.
